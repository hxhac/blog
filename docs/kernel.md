```markdown
---

在 Linux 操作系统中，API（应用程序编程接口）和 ABI（应用程序二进制接口）是两个重要的概念。

API 是一组定义了软件组件之间交互的规则和约定的接口。它定义了应用程序可以使用的函数、数据结构和其他编程构造，以及如何使用它们。在 Linux 内核中，API 定义了内核提供给用户空间应用程序的函数和数据结构，以便应用程序可以与内核进行交互。

ABI 是一组定义了二进制接口的规则和约定，用于在不同的二进制模块之间进行交互。它定义了二进制模块（如应用程序、库和内核模块）之间的函数调用约定、数据结构布局和系统调用接口等。ABI 确保了不同模块之间的二进制兼容性，使它们能够正确地交互和运行。

对于 Linux 内核，API 和 ABI 是非常重要的，因为它们定义了内核与用户空间应用程序之间的接口。通过遵循 API 和 ABI，应用程序可以正确地调用内核提供的功能，并与内核进行通信。同时，ABI 的稳定性也确保了二进制兼容性，使得应用程序可以在不同的内核版本上运行，而无需重新编译。

总结起来，API 定义了内核提供给用户空间应用程序的函数和数据结构，而 ABI 定义了内核与应用程序之间的二进制接口规则和约定。
```

```yaml
# TCB 的主要作用是保存和管理线程的上下文信息，以便在线程切换时能够正确地恢复和执行线程。操作系统通过操作 TCB 来实现线程的创建、调度、切换和销毁等功能。
```

```markdown
是的，线程的 TCB（线程控制块）和进程的 TSS（任务状态段）在某种程度上可以类比。

TCB 是用来存储线程的上下文信息的数据结构，包括寄存器状态、堆栈指针、调度信息等。在线程切换时，操作系统会保存当前线程的上下文信息到其对应的 TCB 中，并加载下一个线程的上下文信息，以实现线程的切换。

类似地，TSS 是用来存储进程的上下文信息的数据结构，包括寄存器状态、堆栈指针、段选择子等。在进程切换时，操作系统会保存当前进程的上下文信息到其对应的 TSS 中，并加载下一个进程的上下文信息，以实现进程的切换。

虽然 TCB 和 TSS 都用于存储上下文信息并在切换时进行加载，但它们的具体实现和功能可能有所不同。TCB 通常是线程库或操作系统内核实现的数据结构，用于管理线程的状态和调度，而 TSS 是由处理器提供的硬件支持，用于管理任务的上下文切换。

总的来说，TCB 和 TSS 都是用来存储上下文信息的数据结构，用于在上下文切换时保存和加载相关的状态。它们在不同的上下文切换场景中发挥着类似的作用，但具体实现和使用方式可能有所不同。
```

```markdown
- *读写锁适用于能明确区分读操作和写操作的场景。*
```

```markdown
CPU 上下文切换的过程如下：
1. 当操作系统决定切换到另一个进程时，它会保存当前进程的 CPU 上下文。这包括将寄存器的值保存到当前进程的内存区域中。
2. 操作系统根据调度算法选择下一个要执行的进程，并从其内存区域中恢复其先前保存的 CPU 上下文。
3. 恢复后，操作系统将控制权转移到新进程的程序计数器（PC）指向的位置，从该位置继续执行。

CPU 上下文切换是一项开销较高的操作，因为涉及到保存和恢复大量的状态信息。它发生在多任务操作系统中，用于实现进程间的切换和资源共享，以便为每个进程提供公平的 CPU 时间片，并使系统能够同时运行多个任务。
```

```markdown
根据提供的文档《TLB之flush操作[一]》，以下是几个问题及其回答：

1. **为什么需要维护TLB和页表（PTE）的一致性？**
   - TLB（Translation Lookaside Buffer）作为一种缓存，需要维护与页表（PTE，Page Table Entry）的一致性，因为TLB存储了虚拟地址到物理地址的映射关系，而这些映射关系是由操作系统通过页表来管理的。当页表内容发生变化时，比如页面被换出或映射被解除，相应的TLB条目也需要更新或清除，以确保地址转换的准确性。

2. **TLB的flush操作与普通cache的flush操作有何不同？**
   - TLB的flush操作是清除某个TLB条目，而不是将数据刷到外部页表去同步。这是因为TLB中的数据不会比页表更新，所以不需要执行同步操作。这与普通cache的flush操作不同，普通cache在flush时通常需要将数据写回内存以保持一致性。

3. **为什么在进程进入kernel mode时不需要flush TLB？**
   - 当进程通过系统调用进入kernel mode，内核处理完后再返回user mode时，无论是进程的页表对应的TLB条目还是内核的页表对应的TLB条目，都不需要被flush。这是因为如果flush了整个TLB，内核将面对一个空的TLB，需要一段时间让内核的常用PTE进入TLB后，才能再次发挥TLB的优势。如果仅flush进程的TLB条目，那么在返回user mode时，TLB对进程来说是空的，这会影响性能。

4. **在进程切换时，为什么kernel的TLB条目可以不被flush？**
   - 在Linux内存模型中，用户空间和内核空间所用的虚拟地址不会重叠，因此在发生进程切换时，kernel的页表通常不会有大的变化。现代处理器在TLB条目中设置了一个名为G（Global）的标志位，用来表示那些在系统运行中不常变化的映射关系，比如内核和虚拟化里的hypervisor。当CR4寄存器里的PGE（page global enable）位被置位时，含有G标志的TLB条目就不会被flush。

5. **什么是PCID和ASID，它们在TLB中的作用是什么？**
   - PCID（Process Context Identifiers）和ASID（Address Space Identifiers）是在TLB条目中用来区分不同进程的标识。在x86架构中，PCID用于标识进程，而在ARM架构中，ASID用于同样的目的。这些标识允许TLB区分和管理属于不同进程的页表映射关系，从而避免不同进程间的地址冲突。通过在CR3寄存器中存储当前进程对应的PCID（在x64架构中），并在CR4寄存器中置位PCIDE位来启用PCID，可以实现在进程切换时不需要flush TLB的效果。
```

```markdown
随机访问存储器 RAM 分为两种：静态 RAM（SRAM）和动态 RAM（DRAM）

SRAM(静态 RAM、静态随机存取存储器)

- SRAM 将每个位存储在一个双稳态的存储器单元里，每个单元用六个晶体管电路实现（成本相对高）。双稳态就是电路可以无限期地保持在两个不同的电压配置或状态之一。其他任何状态都是不稳定的——从不稳定状态开始，电路会迅速地转移到两个稳定状态中的一个。
- 由于双稳态特性，SRAM 只要有电，它就会永远保持它的值（抗干扰性强）。

在 CPU 里，通常会有 L1、L2、L3 三层高速缓存

- 每个 CPU 核心都有一块独占的`L1 缓存`，通常分为指令缓存和数据缓存，L1 缓存通常嵌在 CPU 核心的内部
- `L2 缓存`同样是每个 CPU 核心都有，但是不在 CPU 核心内部，所以 L2 缓存的访问速度比 L1 缓存稍慢
- `L3 缓存`通常由多个 CPU 核心共用，尺寸更大，访问速度更慢

DRAM(动态 RAM、动态随机存取存储器)

- DRAM 需要不断地刷新，才能保持数据被存储起来。DRAM 的存储密度更高。
- DRAM 将每个位存储为对一个电容的充电，每个单元由一个电容和一个访问晶体管组成（成本相对低）。与 SRAM 不同，DRAM 存储器单元对干扰非常敏感（抗干扰性弱）。当电容的电压被扰乱之后，它就永远不会恢复了。暴露在光线下会导致电容电压改变。
- 内存系统必须周期性地通过对 DRAM 读出，然后重写来刷新内存每一位。
```

怎么查看一个进程究竟占用多少内存？

- USS(`Unique Set Size进程独占的物理内存（不包含共享库占用的内存）`) 是一个进程所占用的私有内存。即该进程独占的内存。*USS 是非常非常有用的数据，因为它反映了运行一个特定进程真实的边际成本（增量成本）*。当一个进程被销毁后，USS 是真实返回给系统的内存。当进程中存在一个可疑的内存泄露时，USS 是最佳观察数据。
- PSS(`Proportional Set Size实际使用的物理内存（比例分配共享库占用的内存）`) 与 RSS 不同，它按比例表示使用的共享库，例如：如果有三个进程都使用了一个共享库，共占用了 30 页内存。那么 PSS 将认为每个进程分别占用该共享库 10 页的大小。*PSS 是非常有用的数据，因为系统中所有进程的 PSS 都相加的话，就刚好反映了系统中的总共占用的内存*。而当一个进程被销毁之后，其占用的共享库那部分比例的 PSS，将会再次按比例分配给余下使用该库的进程。这样 PSS 可能会造成一点的误导，因为当一个进程被销毁后，PSS 不能准确地表示返回给全局系统的内存（the memory returned to the overall system）。
- RSS(`Resident Set size常驻内存（包含共享库占用的内存）`) 是一个进程在 RAM 中真实存储的总内存。但是 RSS 还是可能会造成误导，因为它仅仅表示该进程所使用的所有共享库的大小，它不管有多少个进程使用该共享库，该共享库仅被加载到内存一次。所以 RSS 并不能准确反映单进程的内存占用情况。
- VSS(`Virtual Set Size虚拟内存（包含共享库占用的内存）`) 单个进程全部可访问的地址空间，其大小可能包括还尚未在内存中驻留的部分。对于确定单个进程实际内存使用大小，VSS 用处不大。


*结论，VSS>=RSS>=PSS>=USS*

```markdown
我认为把TCB类比成公司二级项目的项目经理很贴切，那么能否用这个类比给我描述一下，整个thread切换上下文的过程？
---

使用公司一级项目项目经理的类比来描述进程上下文切换的过程，我们可以这样想象：

在一个大型公司中，有多个一级项目同时进行，每个项目都有自己的项目经理（类比为PCB，即进程控制块）。这些项目经理负责确保他们各自的项目按照计划和目标推进。当需要在不同项目之间切换注意力时，公司会进行一系列的步骤来确保平滑过渡，这个过程类似于操作系统中的进程上下文切换。

1. **保存当前状态**: 当公司决定从一个项目（进程）切换到另一个项目时，当前项目的项目经理需要记录下项目的所有关键信息。这包括项目进度、下一步计划、当前遇到的问题、团队成员的任务分配等。在进程切换中，操作系统会保存当前进程的状态到其PCB中，包括程序计数器、寄存器状态、内存分配、打开的文件描述符等。

2. **评估和优先级**: 公司会根据项目的紧急程度、截止日期和资源需求来决定下一个要关注的项目。类似地，在操作系统中，进程调度器会根据进程的优先级、调度策略和其他因素来选择下一个要运行的进程。

3. **资源分配**: 确定下一个项目后，公司会重新分配必要的资源，如资金、人力和其他物资。在进程切换中，操作系统会确保新选中的进程有足够的资源来运行，包括内存分配、I/O权限等。

4. **加载新状态**: 接下来，公司将注意力和资源转移到新的项目上，新的项目经理会更新自己的工作状态，准备开始新的工作。在操作系统中，进程调度器会从新选中的进程的PCB中加载状态信息，恢复进程的执行环境。

5. **执行新任务**: 最后，新的项目经理（进程）开始执行他们的任务，推进项目进度。在操作系统中，新进程开始执行其代码，直到完成或再次被操作系统挂起。
```

```markdown
---

MESI（）协议是一种用于保证多核 CPU 缓存一致性的协议。它定义了缓存行的四种状态，并通过协议规定了缓存行状态的转换和处理方式，以确保多核 CPU 之间的缓存数据一致性。

1. Modified（修改）：某个成员对工作材料进行了修改，并且还没有将修改的内容通知给其他成员。

2. Exclusive（独占）：某个成员独占了工作材料，并且没有被其他成员缓存。

3. Shared（共享）：多个成员共享同一份工作材料，并且都没有对其进行修改。

4. Invalid（无效）：某个成员的工作材料无效，即不可用或已过期。

在这个类比中，团队成员之间需要遵循 MESI 协议的规定来保持工作材料的一致性：

- 当一个成员修改了工作材料时，需要将其状态设置为 Modified，并通知其他成员将其缓存行设置为 Invalid，以确保其他成员不使用过期的数据。

- 当一个成员独占了工作材料时，其他成员不能缓存该行数据，以避免冲突和数据不一致。

- 当多个成员共享同一份工作材料时，它们的缓存行状态都设置为 Shared，以确保它们都使用相同的数据。

- 当某个成员不再需要工作材料时，需要将其状态设置为 Invalid，以通知其他成员该行数据已无效。

通过这种类比，我们可以更好地理解 MESI 协议的作用和原理，以及多核 CPU 中缓存一致性的重要性。
```

```markdown
- MESI 只保证 L1-L3 的 cache 之间的内存可见性，但是 CPU 和 L1 之间，还需要 storebuffer 之类的缓存
- *多核处理器的每个核都有自己的 cache，MESI 协议就是用来保证多核的 cache 一致*
- MESI 协议通过回写策略来更新 cache，提高性能的同时带来了额外风险，回写带来的问题可以通过内存屏障解决


### 内存屏障的由来？

针对 store buffer 和 invalidate queue 这两个优化带来的问题，我们又提供了内存屏障作为解决方案。内存屏障交给了编写程序的人的手里，利用它就可以规避上面提到的问题。

内存屏障分为写屏障和读屏障，编写程序时可以在期望的地方加入内存屏障。写屏障会强制 cpu 清空 store buffer 的内容，也就是将所有的变更都写入 cache，随后变更也就写入了内存，使其对其他 cpu 可见；读屏障会强制 cpu 执行 invalidate queue 中的所有 invalidate 操作，使自身的 cache 内容失效，从而使 cpu 从内存或者其他 cpu 中获取最新的 cache 数据。
```

```markdown
4. 内存映射：Linux 支持将文件映射到内存，以便进程可以像访问内存一样访问文件。这种机制被广泛用于加速文件访问和共享内存。
```

```markdown
根据提供的三篇文档，以下是几个问题及其回答：

1. **Linux虚拟地址空间布局是如何的？**
   - 在Linux系统中，每个进程都有自己的虚拟地址空间，通常在32位模式下是4GB的内存地址块。内核空间和用户空间的虚拟内存比例是1:3。虚拟地址通过页表映射到物理内存，页表由操作系统维护。内核空间在页表中拥有较高特权级，用户态程序试图访问这些页时会导致页错误。用户模式地址空间的映射随进程切换的发生而不断变化。

2. **一个程序最多能使用多少内存？**
   - 一个程序能使用的内存量取决于其运行的操作系统和硬件架构。在64位系统中，理论上可以寻址的虚拟内存空间非常大（通常是2的64次方个地址），但实际上受限于CPU的处理能力和物理内存的大小。通常操作系统会允许进程使用非常大，但不到理论最大值的地址空间。例如，现代操作系统可能允许进程使用几十到几百EB（1EB = 10^6TB）的虚拟地址空间。

3. **什么是虚拟内存，它是如何工作的？**
   - 虚拟内存是一种内存管理技术，它允许操作系统设计理论上无限大的地址空间，而实际的物理内存可能有限。操作系统将虚拟内存分成许多小块，称为页（Page），并将这些页映射到物理内存中的Frame。每个进程都有自己的页表，用于维护虚拟地址到物理地址的映射。当进程访问内存时，CPU将虚拟地址发送给内存管理单元（MMU），MMU查询页表，并将虚拟地址转换为物理地址，然后执行内存操作。

4. **什么是大页面问题，它是如何被解决的？**
   - 大页面问题指的是当应用程序需要大量连续的内存时，如果使用小页面（如4KB），则可能需要大量的页表条目来管理这些页面，这会增加内存开销。为了解决这个问题，可以使用大页面（如2MB或1GB），这样可以减少页表的大小，降低内存管理的开销。操作系统和硬件需要支持大页面，这通常通过多级页表来实现。

5. **内存管理单元（MMU）的作用是什么？**
   - 内存管理单元（MMU）是CPU中的一个小型设备，负责将虚拟地址转换为物理地址。它通过查询页表来完成这个转换，并帮助CPU操作真实的物理内存。MMU还负责维护CPU缓存，确保内存访问的效率。
```

```markdown
CPU 如何访问内存

CPU 怎么访问内存？我们回想一下，之前讲过的高级语言和低级语言转化过程。

我们先思考一下，C 语言把我们写出来的变量和函数都转换成了什么呢？如果记不太清了，可以回顾[第十五节课]。没错，C 语言把变量名和函数名都转换成了汇编语言里的标号，而汇编语言的标号，就是机器更好理解的符号。符号具体包括符号类型、符号名称和符号地址这几个属性。其中，符号地址是由一个叫链接器的东西生成的。

汇编语言的标号，就表示为一段内存地址的开始。再根据我们 RISC-V 里访存指令的操作（可回顾[第二十一节课]和[第二十二节课]），进一步分析看看：加载字指令 lw 指令，它会从一个地址指向的内存单元中，把数据加载到寄存器上；储存字指令 sw 指令则是跟 lw 指令相反，把寄存器里的数据存到特定内存单元当中。这些指令里源寄存器和立即数组成的数据，其实就叫内存地址。

结合这些信息，我们就能推出这个结论：CPU 正是通过内存地址来访问内存的。这个地址本质上是一个整数数据。而这个整数代表了一个内存单元的索引号，CPU 访问内存的时候，硬件的地址译码器会负责把索引号，转换成相应的地址信号和片选信号，帮助 CPU“寻路”，找到特定的内存单元位置。

...

现在，我们已经清楚地知道了，CPU 把一个整数数据当成地址，放在地址总线上，由地址译码器选择该地址正确索引的设备或者内存进行访问。

从另一个角度看，数据在物理电路上，是由不同的电子信号的组合来表示的。正是有了这些电子信号组合，才能做到索引相应的设备和内存。CPU 通过什么访问内存，以及地址的本质是什么，我们已经找到答案了，用一句话概括就是：CPU 通过地址访问内存，地址的本质是整数数据，而整数数据的本质是电子信号的组合。
```

```markdown
不同的园区工位的安排和会议室的布局各不相同。

第一种情况是，CPU 小伙伴们坐在一起，会议室在楼层的另一面，大家到会议室里面去都要通过统一的过道，优点简单，缺点是通道会成为瓶颈。
```

```markdown
## 零拷贝


它的主要目的是减少数据拷贝的次数，以提高 I/O 操作的效率。通过使用零拷贝技术，可以避免将数据从内核空间复制到用户空间，从而减少了内存和 CPU 的开销

### 啥是 linux 的拷贝？为啥需要零拷贝？啥是零拷贝？(直接复制的)

拷贝是 linux 中的标准 io 接口的基本数据拷贝的操作，而这一操作会导致数据在 OS 内核地址空间的缓冲区和应用程序地址空间定义的缓冲区之间进行传输

---

- 传统的 IO 操作有性能问题，我们通过`零拷贝`进行改进
- *零拷贝的意思是避免数据拷贝，而不是没有拷贝*

具体来说就是：

- 避免操作系统内核缓冲区之间进行数据拷贝操作。
- 避免操作系统内核和用户应用程序地址空间这两者之间进行数据拷贝操作。
- 用户应用程序可以避开操作系统直接访问硬件存储。
- 数据传输尽量让 DMA 来做。
- 避免不必要的系统调用和上下文切换。
- 需要拷贝的数据可以先被缓存起来。
- 对数据进行处理尽量让硬件来做。

---

零拷贝的方法 (没看懂)

- `直接 I/O`，应用程序直接访问硬件存储，内核只是辅助数据传输，*避免`内核地址空间`的缓冲区和`用户应用程序地址空间`的缓冲区之间的拷贝*
- `DMA+mmap`，数据先通过 DMA 拷贝到内核缓冲区，然后，应用程序和 OS 共享这个缓冲区，这样，内核和应用程序存储空间就不需要再进行任何数据拷贝操作了
- `sendfile`，利用 DMA 将文件中的数据，拷贝到 OS 内核缓冲区，然后将数据拷贝到与 socket 相关的内核缓冲区中，然后，DMA 将数据从内核 socket 缓冲区中拷贝到协议中
- `DMA 辅助的 senfile`，之前我们把页缓存的数据拷贝到 socket 中，实际上，只需要把缓冲区 fd 传到 socket 缓冲区，再把数据长度传过去，这样 DMA 控制器直接将页缓存中的数据打包发送到网络中就可以了

### COW 是什么？写时复制和强制分裂？具体聊聊 linux 中的 COW？

- COW 是一种通用的优化策略，`linux 的 vmm 虚拟内存管理器`，`数据库的很多应用如 snapshot 快照`，以及`各种开发语言`都使用到这种策略，多个调用者在读取时都共享同一份资源，用开发语言举个例子，假设有变量 a 和 b，a=b 做了赋值，两个变量会指向同一个内存地址，该对象的 rc 就是 2；当我们对 b 重新赋值，就发生了`写时复制`，在内存中新建一个内存地址，两个对象的 rc 都是 1
- linux 中 cow 的 fork()；创建一个子进程，内核只为子进程创建虚拟空间，不分配物理内存，和父进程共享物理空间；当父进程中有更改相应段的行为发生时，才为子进程分配物理空间
- cow 在需要写入时，数据才会被复制；没有数据写入时，fork 的开销实际只是复制父进程的页表以及给子进程创建唯一的进程描述符；有数据要写入时，会将要改变的数据页复制给子进程
- 一般把这种被共享访问的页面标记为只读。当一个 task 试图向内存中写入数据时，内存管理单元（MMU）抛出一个异常，内核处理该异常时为该 task 分配一份物理内存并复制数据到此内存，重新向 MMU 发出执行该 task 的写操作
- rc 存在的意义，就是为了 cow 可以正常工作，从而实现对内存的优化
```

```markdown
Linux 并不是所有情况下都是零拷贝的。零拷贝是一种优化技术，旨在减少数据在内存和设备之间的复制次数。在传统的 I/O 操作中，数据需要从用户空间复制到内核空间，然后再从内核空间复制到设备。而在零拷贝操作中，数据可以直接从用户空间传输到设备，减少了中间的数据复制过程。

在 Linux 中，有一些场景仍然需要进行传统的 I/O 操作，而不是零拷贝。例如，当使用标准的文件 I/O 函数（如 read 和 write）时，数据仍然需要在用户空间和内核空间之间进行复制。另外，一些网络协议栈的实现中，也可能需要进行数据的复制操作。

需要注意的是，Linux 提供了一些零拷贝的机制，如 sendfile 和 splice 系统调用，可以在某些情况下实现零拷贝操作。但并不是所有的 I/O 操作都可以使用这些机制，具体取决于应用程序的需求和操作系统的支持。

总的来说，零拷贝是一种优化技术，可以提高 I/O 操作的性能。但并不是所有的情况下都可以使用零拷贝，有些场景仍然需要进行传统的 I/O 操作。
```

```markdown
操作系统虚拟内存中的四种典型页替换算法(OPT,LRU,FIFO,Clock)
```

```markdown
虚拟内存是现代操作系统中的一项关键技术，它允许程序在运行时使用比物理内存更多的内存空间。为了管理虚拟内存，操作系统需要决定哪些内存页应该保留在物理内存中，哪些可以暂时移出到磁盘上的交换空间（swap space）。当CPU需要访问的内存页不在物理内存中时，会发生页面错误（page fault），操作系统必须从磁盘中将该页调入内存。同样，当物理内存不足以容纳所有需要的页时，操作系统必须选择一些页将其移出内存，以便为新的页腾出空间。这个过程称为页替换（page replacement）。

以下是四种典型的页替换算法：

1. **最佳页替换算法（OPT, Optimal Page Replacement）**
   最佳页替换算法是一种理想化的算法，它的核心思想是在发生页面错误时，总是选择将来最长时间不会被再次访问的页进行替换。这意味着，如果操作系统知道每个页在未来会被访问的时间，它可以选择那个最久不会被用到的页来替换。OPT算法是最优的，因为它最小化了页面错误的次数，但它也是不可实现的，因为在实际中我们无法预知未来的访问模式。

2. **最近最少使用算法（LRU, Least Recently Used）**
   LRU算法的工作原理是，当需要替换页时，选择最长时间未被访问的页。这个算法基于一个观察结果：最近使用过的页很可能在不久的将来再次被使用。因此，LRU算法通过跟踪页的访问时间来做出替换决策。实现LRU通常需要一个链表或其他数据结构来记录页的访问顺序。

3. **先进先出算法（FIFO, First-In, First-Out）**
   FIFO算法按照页进入物理内存的顺序来替换它们。最先进入内存的页将是第一个被替换的。这种算法的实现非常简单，只需要一个队列来按顺序存储页即可。然而，FIFO并不总是有效的，因为它不考虑页的访问频率，有时会导致频繁使用的页被替换。

4. **时钟算法（Clock Algorithm）**
   时钟算法是一种近似LRU的算法。它使用一个循环队列（或称为时钟）来存储所有的页框。每个页框都有一个使用位（或称为标记位）。当需要替换页时，时钟算法会遍历队列，检查每个页的使用位。如果位为假，则将其替换；如果位为真，则跳过该页并继续遍历。当遍历完整个队列后，所有位都会被设置为假，这时再次开始遍历。这种算法通过这种方式近似地模拟了LRU算法，同时保持了较低的实现复杂度。

这些算法各有优缺点，并且在不同的工作负载和访问模式下表现不同。在实际的操作系统中，通常会根据具体的需求和环境选择或设计合适的页替换策略。
```

```markdown
---
在虚拟内存系统中，页置换算法（scheduler）的对象是内存中的“页框”（page frames）或者称为“物理页”（physical pages）。这些页框是物理内存的一部分，用于存储从磁盘上的交换空间（swap space）或者从虚拟内存地址空间映射过来的数据。

每个进程都有一个虚拟内存空间，这个空间被分割成许多固定大小的单元，称为“页”（pages）。当进程运行时，它可能会访问其虚拟内存空间中的任何页。如果所需的页不在物理内存中，操作系统必须从磁盘中将其加载到一个空闲的页框中。如果物理内存已满，操作系统必须使用页置换算法来决定哪个页框应该被替换，以便为新的页腾出空间。

因此，在这个上下文中，“页框”可以被视为操作系统调度的对象，就像进程调度中的“进程”一样。操作系统通过页置换算法来管理这些页框，确保物理内存的有效利用，并尽可能减少页面错误（page faults）的发生。这样，虚拟内存系统就可以让每个进程都认为自己拥有连续的、足够的内存空间，即使实际的物理内存可能非常有限。
```

### DMA



```markdown
如果我们对于 I/O 的操作，都是由 CPU 发出对应的指令，然后等待 I/O 设备完成操作之后返回，那 CPU 有大量的时间其实都是在等待 I/O 设备完成操作。

但是，这个 CPU 的等待，在很多时候，其实并没有太多的实际意义。我们对于 I/O 设备的大量操作，其实都只是把内存里面的数据，传输到 I/O 设备而已。在这种情况下，其实 CPU 只是在傻等而已。特别是当传输的数据量比较大的时候，比如进行大文件复制，如果所有数据都要经过 CPU，实在是有点儿太浪费时间了。

因此，计算机工程师们，就发明了 DMA 技术，也就是直接内存访问（Direct Memory Access）技术，来减少 CPU 等待的时间。
```

DMA是用来减少CPU等待时间的，其实可以理解为DPDK之于普通的kernel network


```markdown
其实 DMA 技术很容易理解，本质上，DMA 技术就是我们在主板上放一块独立的芯片。在进行内存和 I/O 设备的数据传输的时候，我们不再通过 CPU 来控制数据传输，而直接通过DMA 控制器（DMA Controller，简称 DMAC）。这块芯片，我们可以认为它其实就是一个协处理器（Co-Processor）。

DMAC 最有价值的地方体现在，当我们要传输的数据特别大、速度特别快，或者传输的数据特别小、速度特别慢的时候。
```

```markdown
- 因为基于`红黑树`、`双向链表`、`事件回调机制`
- 如果有 100 万用户同时与一个进程保持 TCP 连接，但是只有几十个 TCP 连接是活跃的（保持长连接但是未发送 TCP 包）
```

epoll 和 select 实际上没有太大区别，只是 select 用数组标识 fds，epoll 用另一个 epollfd 来标识 fds；（*所以区别只在于 select 的 fds 每次等待都要设置进去，epoll 的 fds 是一开始就设置好的，等待时不需要重新指定*）

> epoll 并不仅仅基于红黑树。它还使用了其他数据结构来管理事件和回调函数，以及内核态和用户态之间的数据传输。epoll 使用红黑树来快速定位需要处理的文件描述符，然后进行相应的 I/O 操作。

epoll 的数据结构称为 eventpoll，是基于红黑树和双线链表的混合结构，使用红黑树来管理 fd，使用双线链表来管理事件和回调函数。也是这种混合结构的设计能让 epoll 多路复用处理并发事件。

```markdown
具体来说，eventpoll 使用红黑树来高效地管理需要监听的文件描述符（file descriptor），以支持快速的插入、删除和查找操作。每个文件描述符都对应一个节点在红黑树中。

而双向链表则用于管理事件和回调函数。在每个红黑树节点中，都有一个链表，用于存储与该文件描述符相关的事件和回调函数。这样，当有事件发生时，可以快速遍历链表，执行相应的回调函数。

因此，可以说 epoll 在内核中使用了双向链表来管理事件和回调函数，而红黑树则用于高效地管理文件描述符。这种混合结构的设计使得 epoll 能够高效地处理大量的并发事件。
```

```markdown
### *select、poll、epoll 分别是什么？有什么区别？*


- select
  - `默认只能监听 1024 个 socket 连接`: 因为每次调用 select 都需要将进程加入到所有监视 socket 的等待队列，每次唤醒都需要从每个队列中移除。这里涉及了两次遍历，而且每次都要将整个 fds 列表传递给内核，有一定的开销。正是因为遍历操作开销大，出于效率的考量，才会规定 select 的最大监听数量，默认只能监听 1024 个 socket。
  - `需要主动轮询`: select 监听 socket 时，不能知道哪个 socket 有可用数据，需要轮询判断。进程被唤醒后，用户程序并不知道哪些 socket 已经接收到数据，还需要遍历一次。
- poll: 与 select 原理类似，只是使用`链表`实现，没有了 socket 限制。

  - `没有 socket 数量限制`: 可以直接知道哪个 socket 有数据了，全面解决了 select 的问题。
  - `线程安全`: epoll 是线程安全的，而 select 和 poll 不是。
```

```markdown
  - **filter表**：这是最主要的安全区域，负责处理所有数据包的过滤决策，类似于公司内部的安全检查站，确保只有符合规定的人员和物品才能进入。
  - **nat表**：负责网络地址转换，可以类比为公司的外事接待部门，处理来访者的注册和身份转换，确保外部人员能够以正确的身份进入公司。
  - **mangle表**：处理数据包的修改和特殊标记，类似于物流部门，对包裹进行分类和标记，以便正确分发。
  - **raw表**：用于设置那些不进行标准处理的数据包的规则，可以类比为特殊事务处理中心，处理一些不按常规流程操作的特殊情况。

2. **五链（Chains）**：
 - 五链可以类比为各个安全区域中的具体工作流程和指引路径。每个链代表了数据包在通过表格时需要经过的特定检查点或处理步骤，类似于：
  - **INPUT链**：检查进入公司的数据包，类似于公司入口的安检，确保所有进入的人员和物品都是安全的。
  - **OUTPUT链**：检查从公司出去的数据包，类似于公司出口的检查，确保没有未授权的物品被带出。
  - **FORWARD链**：处理经过公司的数据包，但不在公司内部停留的，类似于中转站，确保中转的包裹安全。
  - **POSTROUTING链**：修改已经通过nat表的数据包，类似于外事接待部门的后续工作，确保身份转换后的来访者能够顺利离开。
  - **PREROUTING链**：在数据包进入nat表之前进行检查，类似于预检，确保所有即将进入外事接待部门的来访者都符合基本要求。

通过这样的类比，我们可以更直观地理解iptables中的四表五链是如何协同工作，以实现复杂的网络安全和流量管理功能的。
```

```markdown
4. 如果是我的，上层处理完之后，一般会返回一个处理结果，这个处理结果会发出去，称为 output
```

*物理层：因为基于物理电路，所以需要一个将数据转化为物理信号的层*

数据链路层的主协议是 ARP 协议 *用来实现 IP 地址到 MAC 地址的映射* (物理地址（MAC 地址）是数据链路层和物理层使用的地址，IP 地址是网络层和以上各层使用的地址，是一种逻辑地址)

*网络层负责`ip 数据包的产生`以及`ip 数据包在逻辑网络上的路由转发`。网络层的主协议是 IP 协议。* 网络层的其他协议均为 IP 协议的附属协议，比如`BGP 等路由协议族`保证最短路径。`ICMP/IGMP`则保证包在网络上的正常传输 (给传输层以保障，如果错误则直接返回错误)。

*网络层已经解决了基本的发送数据包的问题，传输层的目的则是，针对不同的需求，产生了不同的传输层协议*（比如有一些特殊的需求，有的人想要连接快，不介意丢包，有的人想要连接稳定）。

*传输层已经根据各种需求，封装了不同的传输层协议*。应用层将需求细化，根据我们的实际使用封装了各种供我们直接使用的协议。

可以用“邮局收发邮件来通信”来类比网络中数据包的收发。具体来说，向顺丰下单（第一次请求），顺丰接单（应答)，你向快递小哥联系 (回应应答)，你将消息放进盒子里 (开始封装请求，会话层)，快递小哥封装 一层盒子贴上快递单带回网店 (传输层），到快递点检查是否区域快件（网络层)，将 快件交给运输车 (链路层)，各个快递转运 中心（物理层），快件到达收件市转运中心 (物理层），转运输车（链路层），到达区 域分发（网络层)，网点派送（传输层)，快递员方面签收（会话层），拆开检查（表示层)，收到快递（应用层）。总结一下，下单、揽件、运输、派送、签收，就可以分别看作是应用层、传输层、网络层、链路层、物理层。

- EC 差错控制 (乱序问题) (checksum, seq, ack)
- FC 流量控制 (wnd, tag flow,)
- CC 拥塞控制 (packet-loss based, cwnd, ssthresh, slow start, fast retransmit)
- ARQ 机制 (丢包重传) (Fast Retransmit, After 3 DACK) 丢包重传机制和拥塞控制无关，二者都属于为了保证 TCP 可靠性的机制。丢包重传机制通常会在网络拥塞时触发，所以通常会被误认为是拥塞机制的一部分，这种认知是错误的。总结来说，专人专事，拥塞控制是用来判断网络是否拥塞的，而丢包重传是用来确保数据的可靠传输，而不是主动控制网络拥塞。


如果拿微信发消息举例，EC就是首先得确保你发送的消息没有错别字、语法错误或者信息不连贯，没有颠三倒四、前言不搭后语吧。差错控制通常通过使用校验和、循环冗余校验（CRC）等技术来实现。

- CC则是限制发送方发微信的总量，不让发送方一次发送太多（如果发送太多，就随机丢弃一些信息）。
- FC则是来匹配发送方和接收方的处理信息能力（网络好、接收方能力强就多发，否则就少发），所以需要给重要信息打个 tag，优先处理。
- 滑动窗口：是一种用于流量控制和拥塞控制的技术，它允许发送方在等待确认之前发送一定数量的数据。在微信聊天中，这可以想象为一个“缓冲区”，里面包含了你已经发送但还未得到确认的消息。如果缓冲区满了，你就需要等待一些消息得到确认后再继续发送。
- 拥塞窗口：是拥塞控制中的一个概念，它限制了发送方在任何给定时间可以发送的数据量。在微信聊天的比喻中，这相当于你根据网络状况和对方的接收能力来调整你一次能发送的消息数量。

```markdown

你先在浏览器里面输入 https://www.kaola.com，这是一个 URL。浏览器只知道名字是“www.kaola.com”，但是不知道具体的地点，所以不知道应该如何访问。于是，它打开地址簿去查找。可以使用一般的地址簿协议 DNS 去查找，还可以使用另一种更加精准的地址簿查找协议 HTTPDNS。

无论用哪一种方法查找，最终都会得到这个地址：106.114.138.24。这个是 IP 地址，是互联网世界的“门牌号”。

知道了目标地址，浏览器就开始打包它的请求。对于普通的浏览请求，往往会使用 HTTP 协议；但是对于购物的请求，往往需要进行加密传输，因而会使用 HTTPS 协议。无论是什么协议，里面都会写明“你要买什么和买多少”。

DNS、HTTP、HTTPS 所在的层我们称为应用层。经过应用层封装后，浏览器会将应用层的包交给下一层去完成，通过 socket 编程来实现。下一层是传输层。传输层有两种协议，一种是无连接的协议 UDP，一种是面向连接的协议 TCP。对于支付来讲，往往使用 TCP 协议。所谓的面向连接就是，TCP 会保证这个包能够到达目的地。如果不能到达，就会重新发送，直至到达。

TCP 协议里面会有两个端口，一个是浏览器监听的端口，一个是电商的服务器监听的端口。操作系统往往通过端口来判断，它得到的包应该给哪个进程。

传输层封装完毕后，浏览器会将包交给操作系统的网络层。网络层的协议是 IP 协议。在 IP 协议里面会有源 IP 地址，即浏览器所在机器的 IP 地址和目标 IP 地址，也即电商网站所在服务器的 IP 地址。

操作系统既然知道了目标 IP 地址，就开始想如何根据这个门牌号找到目标机器。操作系统往往会判断，这个目标 IP 地址是本地人，还是外地人。如果是本地人，从门牌号就能看出来，但是显然电商网站不在本地，而在遥远的地方。

操作系统知道要离开本地去远方。虽然不知道远方在何处，但是可以这样类比一下：如果去国外要去海关，去外地就要去网关。而操作系统启动的时候，就会被 DHCP 协议配置 IP 地址，以及默认的网关的 IP 地址 192.168.1.1。

操作系统如何将 IP 地址发给网关呢？在本地通信基本靠吼，于是操作系统大吼一声，谁是 192.168.1.1 啊？网关会回答它，我就是，我的本地地址在村东头。这个本地地址就是 MAC 地址，而大吼的那一声是 ARP 协议。


城关与城关之间是一个国家，当网络包知道了下一步去哪个城关，还是要使用国家内部的 MAC 地址，通过下一个城关的 MAC 地址，找到下一个城关，然后再问下一步的路怎么走，一直到走出最后一个城关。

最后一个城关知道这个网络包要去的地方。于是，对着这个国家吼一声，谁是目标 IP 啊？目标服务器就会回复一个 MAC 地址。网络包过关后，通过这个 MAC 地址就能找到目标服务器。

目标服务器发现 MAC 地址对上了，取下 MAC 头来，发送给操作系统的网络层。发现 IP 也对上了，就取下 IP 头。IP 头里会写上一层封装的是 TCP 协议，然后将其交给传输层，即 TCP 层。
```

```markdown
1.距离矢量路由算法
第一大类的算法称为距离矢量路由（distance vector routing）。它是基于 Bellman-Ford 算法的。

这种算法的基本思路是，每个路由器都保存一个路由表，包含多行，每行对应网络中的一个路由器，每一行包含两部分信息，一个是要到目标路由器，从那条线出去，另一个是到目标路由器的距离。

由此可以看出，每个路由器都是知道全局信息的。那这个信息如何更新呢？每个路由器都知道自己和邻居之间的距离，每过几秒，每个路由器都将自己所知的到达所有的路由器的距离告知邻居，每个路由器也能从邻居那里得到相似的信息。

2.链路状态路由算法
第二大类算法是链路状态路由（link state routing），基于 Dijkstra 算法。

这种算法的基本思路是：当一个路由器启动的时候，首先是发现邻居，向邻居 say hello，邻居都回复。然后计算和邻居的距离，发送一个 echo，要求马上返回，除以二就是距离。然后将自己和邻居之间的链路状态包广播出去，发送到整个网络的每个路由器。这样每个路由器都能够收到它和邻居之间的关系的信息。因而，每个路由器都能在自己本地构建一个完整的图，然后针对这个图使用 Dijkstra 算法，找到两点之间的最短路径。

不像距离距离矢量路由协议那样，更新时发送整个路由表。链路状态路由协议只广播更新的或改变的网络拓扑，这使得更新信息更小，节省了带宽和 CPU 利用率。而且一旦一个路由器挂了，它的邻居都会广播这个消息，可以使得坏消息迅速收敛。
```

```markdown
1.基于链路状态路由算法的 OSPF
OSPF（Open Shortest Path First，开放式最短路径优先）就是这样一个基于链路状态路由协议，广泛应用在数据中心中的协议。由于主要用在数据中心内部，用于路由决策，因而称为内部网关协议（Interior Gateway Protocol，简称 IGP）。

内部网关协议的重点就是找到最短的路径。在一个组织内部，路径最短往往最优。当然有时候 OSPF 可以发现多个最短的路径，可以在这多个路径中进行负载均衡，这常常被称为等价路由。

这一点非常重要。有了等价路由，到一个地方去可以有相同的两个路线，可以分摊流量，还可以当一条路不通的时候，走另外一条路。这个在后面我们讲数据中心的网络的时候，一般应用的接入层会有负载均衡 LVS。它可以和 OSPF 一起，实现高吞吐量的接入层设计。

有了内网的路由协议，在一个国家内，唐僧可以想怎么走怎么走了，两条路选一条也行。

2.基于距离矢量路由算法的 BGP
但是外网的路由协议，也即国家之间的，又有所不同。我们称为外网路由协议（Border Gateway Protocol，简称 BGP）。

在一个国家内部，有路当然选近的走。但是国家之间，不光远近的问题，还有政策的问题。例如，唐僧去西天取经，有的路近。但是路过的国家看不惯僧人，见了僧人就抓。例如灭法国，连光头都要抓。这样的情况即便路近，也最好绕远点走。

对于网络包同样，每个数据中心都设置自己的 Policy。例如，哪些外部的 IP 可以让内部知晓，哪些内部的 IP 可以让外部知晓，哪些可以通过，哪些不能通过。这就好比，虽然从我家里到目的地最近，但是不能谁都能从我家走啊！

在网络世界，这一个个国家成为自治系统 AS（Autonomous System）。自治系统分几种类型。

Stub AS：对外只有一个连接。这类 AS 不会传输其他 AS 的包。例如，个人或者小公司的网络。

Multihomed AS：可能有多个连接连到其他的 AS，但是大多拒绝帮其他的 AS 传输包。例如一些大公司的网络。

Transit AS：有多个连接连到其他的 AS，并且可以帮助其他的 AS 传输包。例如主干网。

每个自治系统都有边界路由器，通过它和外面的世界建立联系。

BGP 又分为两类，eBGP 和 iBGP。自治系统间，边界路由器之间使用 eBGP 广播路由。内部网络也需要访问其他的自治系统。边界路由器如何将 BGP 学习到的路由导入到内部网络呢？就是通过运行 iBGP，使得内部的路由器能够找到到达外网目的地的最好的边界路由器。

BGP 协议使用的算法是路径矢量路由协议（path-vector protocol）。它是距离矢量路由协议的升级版。

前面说了距离矢量路由协议的缺点。其中一个是收敛慢。在 BGP 里面，除了下一跳 hop 之外，还包括了自治系统 AS 的路径，从而可以避免坏消息传得慢的问题，也即上面所描述的，B 知道 C 原来能够到达 A，是因为通过自己，一旦自己都到达不了 A 了，就不用假设 C 还能到达 A 了。

另外，在路径中将一个自治系统看成一个整体，不区分自治系统内部的路由器，这样自治系统的数目是非常有限的。就像大家都能记住出去玩，从中国出发先到韩国然后到日本，只要不计算细到具体哪一站，就算是发送全局信息，也是没有问题的。
```

```markdown
那在网络层发送数据包时，会自动选择 OSPF 或者 BGP 吗？是怎么进行选择的呢？

在网络层发送数据包时，通常不会自动选择 OSPF 或 BGP。OSPF 和 BGP 是动态路由协议，用于学习和更新路由表。它们通常在路由器之间运行，并根据网络拓扑和策略来决定最佳路由。当一个路由器接收到数据包时，它会使用路由表来决定将数据包发送到哪个接口。路由表中的路由可以通过静态路由配置手动添加，也可以通过动态路由协议自动学习和更新。

在选择最佳路由时，动态路由协议通常会考虑各种因素，如链路质量、网络拓扑和策略。不同的动态路由协议有不同的算法和策略来选择最佳路由。例如，OSPF 使用链路状态信息来计算最短路径，而 BGP 使用路径属性和策略来选择路由。

所以，路由器通常会根据路由表中的最佳匹配来选择路由，而不是自动选择 OSPF 或 BGP。选择使用哪种路由协议取决于网络的需求、策略和拓扑结构。

总的来说，动态路由提供了更大的灵活性和可靠性，但需要进行计算和交换路由信息。而静态路由则更简单，但需要手动维护。选择使用静态路由还是动态路由取决于网络的规模、复杂程度以及管理要求等因素。

对了，我们还漏掉了 RIP 协议，那 RIP 协议相比于 OSPF 和 BGP 有什么区别和特点呢？

RIP（Routing Information Protocol）是一种距离向量路由协议，主要用于小型网络中。它使用跳数（hop count）作为衡量路径优劣的指标，并通过将路由表信息广播到相邻路由器来实现路由的学习和更新。RIP 协议对网络的规模和拓扑变化的适应性较差，因此在大型网络中使用较少。相比之下，OSPF（Open Shortest Path First）和 BGP（Border Gateway Protocol）是链路状态路由协议，具有更高的灵活性和可扩展性。OSPF 协议使用链路状态数据库和 Dijkstra 算法来计算最短路径，适用于中等规模的企业网络。BGP 协议是一个可实现自治系统之间路由选择的协议，主要用于互联网的外部路由选择。

总的来说，RIP 协议适用于小型网络，使用跳数作为路径选择的度量标准；OSPF 协议适用于中等规模的企业网络，使用链路状态数据库和最短路径算法；BGP 协议适用于互联网的外部路由选择。
```

```markdown
为什么可以把 AS 自治系统理解为负权环呢？

---

将自治系统（AS，Autonomous System）理解为负权环（Negative Weight Cycle）是一种简化的比喻，用于描述 AS 路径选择中的一种情况。

在路由协议中，AS 是互联网中的一个独立管理的网络运营域，它可以包含多个网络设备和路由器。AS 之间通过边界路由器（Border Router）相互连接，并通过路由协议来交换路由信息。

负权环是一个图论概念，指的是在有向加权图中存在一条环路，其路径上的权重之和为负值。在这种情况下，如果一个数据包在该环路上进行无限循环，其累积的权重会不断减小，形成负权环。

在 AS 路径选择中，路由协议（如 BGP）使用了一种叫做"最短路径优先"（Shortest Path First）的算法来选择最佳路径。这个算法会根据路径的权重或者距离来选择最短路径。然而，当存在负权环时，算法可能无法收敛，因为它会不断选择累积权重为负的环路，导致无限循环。

将 AS 自治系统理解为负权环是为了形象地描述这种情况。当 AS 之间存在某种问题或配置错误时，可能会导致路由协议选择了一个负权环，从而导致数据包在网络中无限循环，无法正确路由。

需要注意的是，实际的 AS 路径选择是复杂而动态的，涉及到多个因素和策略。尽管负权环是一种可能的情况，但它并不是 AS 路径选择中的唯一问题。
```

```markdown
如果最后一跳的时候，IP 改变了怎么办？

---

对于 IP 层来讲，当包到达最后一跳的时候，原来的 IP 不存在了。比如网线拔掉了，或者服务器直接宕机了，则 ARP 就找不到了，所以这个包就会发送失败了。对于 IP 层的工作就结束了。

但是 IP 层之上还有 TCP 层，TCP 会重试的，包还是会重新发送，但是如果服务器没有启动起来，超过一定的次数，最终放弃。

如果服务器重启了，IP 还是原来的 IP 地址，这个时候 TCP 重新发送的一个包的时候，ARP 是能够得到这个地址的，因而会发到这台机器上来，但是机器上面没有启动服务端监听那个端口，于是会发送 ICMP 端口不可达。

如果服务器重启了，服务端也重新启动了，也在监听那个端口了，这个时候 TCP 的服务端由于是新的，Sequence Number 根本对不上，说明不是原来的连接，会发送 RST。

那有没有可能有特殊的场景 Sequence Number 也能对的上呢？按照 Sequence Number 的生成算法，是不可能的。

但是有一个非常特殊的方式，就是虚拟机的热迁移，从一台物理机迁移到另外一台物理机，IP 不变，MAC 不变，内存也拷贝过去，Sequence Number 在内存里面也保持住了，在迁移的过程中会丢失一两个包，但是从 TCP 来看，最终还是能够连接成功的。
```

```markdown
组播和广播的意义和原理是什么？

---

在《TCP/IP 详解》这本书里面，有两章讲了广播、多播以及 IGMP。广播和组播分为两个层面，其中 MAC 层有广播和组播对应的地址，IP 层也有自己的广播地址和组播地址。

广播相对比较简单，MAC 层的广播为 ff:ff:ff:ff:ff:ff，IP 层指向子网的广播地址为主机号为全 1 且有特定子网号的地址。

组播复杂一些，MAC 层中，当地址中最高字节的最低位设置为 1 时，表示该地址是一个组播地址，用十六进制可表示为 01:00:00:00:00:00。IP 层中，组播地址为 D 类 IP 地址，当 IP 地址为组播地址的时候，有一个算法可以计算出对应的 MAC 层地址。

多播进程将目的 IP 地址指明为多播地址，设备驱动程序将它转换为相应的以太网地址，然后把数据发送出去。这些接收进程必须通知它们的 IP 层，它们想接收的发给定多播地址的数据报，并且设备驱动程序必须能够接收这些多播帧。这个过程就是“加入一个多播组”。

当多播跨越路由器的时候，需要通过 IGMP 协议告诉多播路由器，多播数据包应该如何转发。
```

```markdown
TIME_WAIT 状态太多是怎么回事儿？

---

如果处于 TIMEWAIT 状态，说明双方建立成功过连接，而且已经发送了最后的 ACK 之后，才会处于这个状态，而且是主动发起关闭的一方处于这个状态。

如果存在大量的 TIMEWAIT，往往是因为短连接太多，不断的创建连接，然后释放连接，从而导致很多连接在这个状态，可能会导致无法发起新的连接。解决的方式往往是：

- 打开 tcp_tw_recycle 和 tcp_timestamps 选项；
- 打开 tcp_tw_reuse 和 tcp_timestamps 选项；
- 程序中使用 SO_LINGER，应用强制使用 rst 关闭。

当客户端收到 Connection Reset，往往是收到了 TCP 的 RST 消息，RST 消息一般在下面的情况下发送：

- 试图连接一个未被监听的服务端；
- 对方处于 TIMEWAIT 状态，或者连接已经关闭处于 CLOSED 状态，或者重新监听 seq num 不匹配；
- 发起连接时超时，重传超时，keepalive 超时；
- 在程序中使用 SO_LINGER，关闭连接时，放弃缓存中的数据，给对方发送 RST。
```

```markdown
1 设备缓存会导致延时？
```

```markdown
这里还有个问题，就是时间漂移问题。尽管我们前面提及了精确测量时间、时间同步协议等尽可能保证时间准确的方式，但是效果也是”尽可能”让其精确，我们还是不能百分百地保证全球所有电子设备、分布式系统中节点的时间是完全一致的，甚至是同一个计算机但是是多处理器系统中的多个时钟也不是完全一致的。

这回带来什么问题呢？如果运行其上的系统，依赖“时间”对操作顺序执行先后做判断，那这样的系统很可能是存在问题的。

在分布式系统设计中，一般会采用“逻辑时钟”、“向量时钟”等方式来代替真实时钟，来作为操作顺序执行先后的判断依据。这里的内容有很多，后面我们有机会再单独写一篇文章来介绍。

```

```markdown

方案                         优点                                                     缺点
逻辑时钟       - 简单易实现<br>- 不依赖于节点之间的通信<br>- 可以检测因果关系 - 无法准确反映物理时间<br>- 不适用于并发冲突的解决和一致性问题
向量时钟       - 可以检测因果关系<br>- 可以解决并发冲突和一致性问题         - 需要维护较大的向量<br>- 通信开销较大<br>- 可能存在复杂的更新逻辑

---

在选择逻辑时钟或向量时钟时，没有绝对的"更好"方法，而是要根据具体的需求和场景来确定。这取决于分布式系统的特点、应用需求和实现复杂性等因素。下面是一些考虑因素：

1. 简单性和易实现性：
 - 逻辑时钟相对简单，只需要维护一个递增计数器，易于实现和理解。
 - 向量时钟需要维护一个向量，实现和维护稍微复杂一些。

2. 物理时间的需求：
 - 逻辑时钟不能准确反映物理时间，只能依靠事件发生的因果关系进行排序。
 - 向量时钟也不能直接提供准确的物理时间，但可以通过比较向量元素来确定因果关系。

3. 应用需求：
 - 逻辑时钟适用于一些不需要考虑并发冲突和一致性问题的场景。
 - 向量时钟更适用于需要解决并发冲突和一致性问题的场景，例如分布式数据库、分布式锁等。

4. 通信开销：
 - 逻辑时钟不需要节点之间的通信，仅依赖于事件发生的因果关系。
 - 向量时钟需要节点之间的通信来传递和更新向量，可能引入一定的通信开销。

需要根据具体需求进行权衡和选择。在某些情况下，逻辑时钟可能更简单和适用，而在需要解决并发冲突和一致性问题的场景中，向量时钟可能更合适。一些系统可能会结合两种方法，根据具体需求来灵活选择使用。

因此，没有一种方法可以被普遍视为更好，而是要根据分布式系统的特点和应用需求来选择适合的时钟方案。
```

```markdown
在计算机系统中，缓存行（Cache Line）是一种硬件优化机制，用于加速处理器与主内存之间的数据传输。它是计算机体系结构中缓存层次结构中的一个重要概念。

缓存行是缓存最小的可分配单位，它是一组连续的字节或字（通常是64个字节），在主内存和缓存之间进行数据传输。当处理器访问主内存中的数据时，它会以缓存行作为基本单位从主内存中加载数据到缓存中。如果处理器需要访问的数据已经在缓存行中，那么它可以直接从缓存中读取，而不需要访问较慢的主内存。

缓存行的大小是由硬件设计决定的，并且在不同的计算机体系结构中可能有所不同。常见的大小是64字节，但也有其他大小的实现。缓存行的大小通常与处理器的字长和内存总线宽度相关。

缓存行的使用有助于减少内存访问的延迟，提高数据传输的效率。当处理器加载缓存行时，它会加载与所需数据相邻的其他数据，以利用空间局部性（Spatial Locality）和时间局部性（Temporal Locality）的原理。这样，在处理器需要访问相邻的数据时，可以直接从缓存中获取，而不需要额外的内存访问。

了解缓存行的大小和工作原理对于进行高性能编程和优化非常重要。在编写代码时，可以通过合理地利用缓存行，最大程度地减少缓存未命中（Cache Miss）和内存访问延迟，提高程序的性能。
```
